# Story 7.3: Claim Profile Button with Verification Prompt - Brownfield Addition

## Status
Ready for Development

## Story
**As a** crew member viewing a profile,
**I want** to see a "Claim Profile" button with information about verification benefits,
**so that** I can easily initiate the verification process to get my profile verified and increase my booking opportunities.

## Story Context

**Existing System Integration:**
- Integrates with: Profile page (`apps/web/app/(public)/crew/[slug]/page.tsx`), ProfileHeader component (`apps/web/components/profile/ProfileHeader.tsx`)
- Technology: Next.js App Router, React Server/Client Components, TypeScript, Tailwind CSS
- Follows pattern: Existing modal pattern from AdminProfileEditModal, existing button styling from ProfileHeader
- Touch points: Profile page UI, ProfileHeader component, Profile type (is_claimed field)

## Acceptance Criteria

**Functional Requirements:**

1. "Claim Profile" button appears on profile page for unclaimed profiles only (when `profile.is_claimed === false`)
2. Button displays text: "Claim Profile" with a subtitle/message: "Verified profiles get booked ~3x more on average"
3. Button is prominently displayed in the ProfileHeader component, visible and accessible
4. Clicking the button opens a modal/popup dialog
5. Modal displays clear instructions: "Email niko.vernic@gmail.com with a picture of your ID to verify"
6. Modal includes a close button or click-outside-to-close functionality
7. Modal styling matches existing design patterns (similar to AdminProfileEditModal)
8. All existing profile functionality continues to work unchanged (contact forms, links, etc.)

**Integration Requirements:**

9. Button only renders for unclaimed profiles (check `profile.is_claimed` field)
10. Modal follows existing modal pattern used in AdminProfileEditModal
11. UI follows existing ProfileHeader component styling and layout patterns
12. No backend changes required - this is a UI-only feature
13. Component is client-side interactive (uses 'use client' directive if needed)

**Quality Requirements:**

14. Change is covered by appropriate tests (component tests for button visibility and modal behavior)
15. No regression in existing profile page functionality verified
16. Code follows existing coding standards and project structure
17. Modal is accessible (keyboard navigation, focus management)

## Technical Notes

- **Integration Approach:**
  - Add "Claim Profile" button to ProfileHeader component
  - Button should be conditionally rendered based on `profile.is_claimed === false`
  - Create a simple ClaimVerificationModal component (client component)
  - Use existing modal pattern from AdminProfileEditModal as reference
  - Place button prominently in ProfileHeader, perhaps after the profile badges or in the contact section area
  - Modal should be a simple overlay with instructions and close button

- **Existing Pattern Reference:**
  - Modal pattern: `apps/web/components/admin/AdminProfileEditModal.tsx` - shows overlay modal structure
  - ProfileHeader: `apps/web/components/profile/ProfileHeader.tsx` - shows button/link styling patterns
  - Profile type: `packages/shared/src/types/index.ts` - Profile interface with `is_claimed` field

- **Key Constraints:**
  - Only show button for unclaimed profiles
  - Keep modal simple - just instructions, no form submission needed
  - Email address: niko.vernic@gmail.com (hardcoded for now)
  - No backend API needed - this is purely informational UI
  - Modal should be dismissible (close button and/or click outside)

## Risk and Compatibility Check

**Minimal Risk Assessment:**

- **Primary Risk:** Button might show for claimed profiles or modal might interfere with existing UI
- **Mitigation:** Proper conditional rendering based on `is_claimed` field, modal uses z-index overlay pattern that won't interfere
- **Rollback:** Simple - remove button component and modal, revert ProfileHeader changes

**Compatibility Verification:**

- [x] No breaking changes to existing APIs (UI-only change)
- [x] Database changes (if any) are additive only (no database changes)
- [x] UI changes follow existing design patterns (ProfileHeader, modal patterns)
- [x] Performance impact is negligible (simple conditional rendering, client-side modal)
