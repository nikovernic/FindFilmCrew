# Story 7.2: Admin Profile Editing and Verification Toggle

## Status
Ready for Review

## Story
**As a** platform administrator,
**I want** to edit all profile information and easily toggle verification status for any profile from the admin dashboard,
**so that** I can maintain data quality, correct errors, and manually verify profiles when needed.

## Story Context

**Existing System Integration:**
- Integrates with: Admin dashboard (`apps/web/app/(auth)/admin/page.tsx`), ProfileService (`apps/web/lib/services/profileService.ts`), Admin API routes (`apps/web/app/api/admin/profiles`)
- Technology: Next.js App Router, React Server/Client Components, Supabase, TypeScript, shadcn/ui
- Follows pattern: Existing admin approve/reject pattern, profile edit form pattern from Story 6.4
- Touch points: Admin dashboard UI, ProfileService.updateProfile(), Admin API routes, Profile database schema

## Acceptance Criteria

**Functional Requirements:**

1. Admin can navigate to and view a list of all profiles (or search/filter profiles) from the `/admin` dashboard page
2. Profile list is accessible as a section/tab on the admin dashboard (alongside existing "Pending Review" section)
3. Admin can click on any profile from the list to view full details and edit form
3. Admin can edit all profile fields: name, role, location, contact info, bio, URLs, union status, experience, secondary roles, additional markets, credits
4. Admin can toggle profile verification status (verified/unverified) with a single click/button
5. When toggling verification, if setting to verified, set `is_verified: true` and `verified_at: current timestamp`
6. When toggling verification, if setting to unverified, set `is_verified: false` and `verified_at: null`
7. Profile updates are saved immediately with success/error feedback
8. All existing profile functionality continues to work unchanged (search, display, claiming, etc.)

**Integration Requirements:**

9. New functionality follows existing admin authentication pattern (`requireAdmin` middleware)
10. Profile editing follows existing ProfileService.updateProfile() pattern
11. UI follows existing admin dashboard design patterns (PendingReviewSection style)
12. API endpoints follow existing admin API route structure (`/api/admin/profiles/[id]`)
13. All functionality is accessible from the `/admin` dashboard page with clear navigation between sections

**Quality Requirements:**

13. Change is covered by appropriate tests (unit tests for service, integration tests for API)
14. No regression in existing functionality verified
15. Code follows existing coding standards and project structure

## Technical Notes

- **Integration Approach:** 
  - Extend admin dashboard at `/admin` page with new "All Profiles" section (alongside existing "Pending Review" section)
  - Add navigation/tabs to switch between "Pending Review" and "All Profiles" sections on the same dashboard page
  - Create profile list/table component showing key profile info (name, role, location, verification status, claim status)
  - Create profile edit modal/page component accessible from profile list (can reuse/edit ProfileEditForm pattern from Story 6.4)
  - Add verification toggle button/switch component prominently in the edit interface
  - Create `PUT /api/admin/profiles/[id]` endpoint for profile updates
  - Create `POST /api/admin/profiles/[id]/verify` or use PUT endpoint with `is_verified` field
  - Extend ProfileService with admin update method (or use existing updateProfile with admin override)

- **Existing Pattern Reference:**
  - Admin authentication: `apps/web/lib/middleware/auth.ts` - `requireAdmin()` function
  - Profile update: `apps/web/lib/services/profileService.ts` - `updateProfile()` method
  - Admin API pattern: `apps/web/app/api/admin/profiles/[id]/approve/route.ts`
  - Profile edit form: `apps/web/components/profile/ProfileEditForm.tsx` (can adapt for admin use)

- **Key Constraints:**
  - Must maintain admin-only access (use `requireAdmin` middleware)
  - Must preserve existing profile update logic for crew members
  - Verification toggle should be immediate (no confirmation needed, but show clear feedback)
  - Profile list should support pagination if there are many profiles

## Risk and Compatibility Check

**Minimal Risk Assessment:**

- **Primary Risk:** Admin update might conflict with crew member profile updates or break existing update logic
- **Mitigation:** Use separate admin update method or extend existing with admin role check, ensure proper validation
- **Rollback:** Revert admin dashboard changes, API endpoints can remain but unused

**Compatibility Verification:**

- [x] No breaking changes to existing APIs (new endpoints only)
- [x] Database changes (if any) are additive only (using existing `is_verified` and `verified_at` fields)
- [x] UI changes follow existing design patterns (admin dashboard, form components)
- [x] Performance impact is negligible (same database operations, just admin-initiated)
